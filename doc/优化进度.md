**降级策略**：
1. Redis连接失败时自动降级到NodeCache
2. Redis重试3次后失败，切换到降级模式
3. 降级后继续尝试重连Redis

**缓存操作**：
- `get<T>(key)` - 获取缓存
- `set<T>(key, value, ttl?)` - 设置缓存
- `del(key)` - 删除缓存
- `delPattern(pattern)` - 批量删除
- `exists(key)` - 检查是否存在
- `incr(key)` - 自增
- `decr(key)` - 自减
- `getTTL(key)` - 获取过期时间
- `setTTL(key, ttl)` - 设置过期时间
- `flushAll()` - 清空所有缓存

**核心功能**：

1. **限流（令牌桶算法）**：
   - 基于IP和路径的限流
   - 支持Redis分布式限流
   - 可配置限流阈值和TTL
   - 支持跳过成功/失败请求

2. **熔断器（三态模型）**：
   - CLOSED（关闭）：正常状态
   - OPEN（打开）：熔断状态
   - HALF_OPEN（半开）：尝试恢复
   - 基于错误率和响应时间触发
   - 支持fallback降级

3. **服务降级（三级降级）**：
   - NORMAL（0级）：正常服务
   - MINIMAL（1级）：最小化服务
   - EMERGENCY（2级）：紧急模式
   - 支持自动降级和恢复
   - 支持全局降级级别控制

4. **分布式锁（Redis实现）**：
   - 基于Redis的SETNX实现
   - 支持锁重试和超时
   - 使用UUID保证锁唯一性
   - 支持锁续期和强制释放

**监控API**：
- `GET /concurrency-control/circuit-breaker/:key` - 获取熔断器状态
- `POST /concurrency-control/circuit-breaker/:key/reset` - 重置熔断器
- `POST /concurrency-control/circuit-breaker/reset-all` - 重置所有熔断器
- `GET /concurrency-control/service-degradation/:serviceName` - 获取服务降级状态
- `GET /concurrency-control/service-degradation` - 获取所有服务降级状态
- `POST /concurrency-control/service-degradation/:serviceName/degrade` - 降级服务
- `POST /concurrency-control/service-degradation/:serviceName/recover` - 恢复服务
- `PUT /concurrency-control/service-degradation/global-level` - 设置全局降级级别
- `GET /concurrency-control/distributed-lock/:key` - 检查分布式锁状态
- `GET /concurrency-control/distributed-lock/active` - 获取所有活跃的分布式锁
- `DELETE /concurrency-control/distributed-lock/:key/force-release` - 强制释放分布式锁
- `GET /concurrency-control/status` - 获取并发控制整体状态

**核心指标**：

1. **HTTP请求指标**：
   - `http_requests_total` - HTTP请求总数（按方法、路径、状态码分类）
   - `http_request_duration_seconds` - HTTP请求响应时间（直方图）
   - `http_requests_in_progress` - 当前正在处理的请求数
   - `http_errors_total` - HTTP错误总数（按错误类型分类）

2. **数据库指标**：
   - `db_connections_active` - 活跃的数据库连接数
   - `db_query_duration_seconds` - 数据库查询响应时间（直方图）

3. **缓存指标**：
   - `cache_hits_total` - 缓存命中总数
   - `cache_misses_total` - 缓存未命中总数

4. **并发控制指标**：
   - `circuit_breaker_state` - 熔断器状态（0=CLOSED, 1=OPEN, 2=HALF_OPEN）
   - `service_degradation_level` - 服务降级级别（0=NORMAL, 1=MINIMAL, 2=EMERGENCY）
   - `distributed_locks_active` - 活跃的分布式锁数量

5. **系统指标**：
   - `system_cpu_usage` - 系统CPU使用率
   - `system_memory_usage` - 系统内存使用率

**监控API**：
- `GET /performance-monitor/metrics` - 获取Prometheus格式指标
- `GET /performance-monitor/metrics/reset` - 重置所有指标
- `GET /performance-monitor/traces/current` - 获取当前追踪上下文
- `GET /performance-monitor/traces/active` - 获取活跃的追踪
- `GET /performance-monitor/traces/completed` - 获取已完成的追踪
- `POST /performance-monitor/traces/clear` - 清除已完成的追踪
- `GET /performance-monitor/alerts/active` - 获取活跃的告警
- `GET /performance-monitor/alerts/history` - 获取告警历史
- `POST /performance-monitor/alerts/resolve` - 解决告警
- `POST /performance-monitor/alerts/clear` - 清除已解决的告警
- `GET /performance-monitor/status` - 获取性能监控状态

**告警规则**：
- High QPS: QPS超过1000触发警告
- High Response Time: 响应时间超过3秒触发警告
- High Error Rate: 错误率超过5%触发严重告警
- High CPU Usage: CPU使用率超过80%触发警告
- High Memory Usage: 内存使用率超过80%触发警告

**告警通知方式**：
- 邮件通知（需配置EmailService）
- Webhook通知（需配置webhook URL）

**链路追踪特性**：
- 基于AsyncLocalStorage实现上下文传递
- 支持父子Span嵌套
- 自动生成traceId和spanId
- 支持标签和日志记录
- 可配置采样率（默认10%）
- 支持追踪历史查询和过滤
