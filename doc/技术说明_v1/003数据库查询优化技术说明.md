# 数据库查询优化技术说明

## 一、背景与问题

在百万级并发场景下，数据库查询性能成为系统的关键瓶颈。传统的数据库访问方式存在以下问题：

1. **慢查询无法及时发现**：缺乏慢查询监控机制，导致性能问题难以定位
2. **重复查询浪费资源**：相同查询反复执行，增加数据库负载
3. **读写分离未实现**：所有操作都走主库，主库压力大
4. **缺乏查询优化策略**：没有统一的查询优化和缓存机制

这些问题导致：
- 数据库响应时间长
- 系统吞吐量低
- 数据库连接池耗尽
- 系统稳定性差

## 二、解决方案

数据库查询优化模块提供了完整的查询优化解决方案，包括：

### 1. 慢查询监控
自动记录执行时间超过阈值的查询，提供统计分析功能。

**核心功能**：
- 自动记录慢查询（默认阈值1000ms）
- 支持按实体类型和操作类型过滤
- 提供查询统计分析（平均时长、最大时长、最小时长）
- 自动清理过期查询（1小时）
- 支持查询执行历史记录

### 2. 查询缓存
基于本地内存和Redis的双层缓存机制，减少重复查询。

**核心功能**：
- 支持本地内存缓存和Redis缓存
- 基于查询语句和参数生成缓存键（MD5哈希）
- 支持自定义TTL（默认300秒）
- LRU缓存淘汰策略
- 支持按查询、按模式失效缓存
- 提供缓存命中率统计
- 自动清理过期缓存

### 3. 数据库读写分离
支持主从数据库分离，降低主库压力。

**核心功能**：
- 支持主从数据库分离
- 可配置读写比例（默认70%读操作走从库）
- 事务操作强制使用主库
- 提供读写统计信息
- 支持强制指定使用主库或从库

### 4. 查询优化拦截器
提供统一的查询优化拦截器，自动应用缓存和监控。

**核心功能**：
- 自动缓存查询结果
- 自动监控慢查询
- 支持Repository级别的查询优化
- 提供查询性能分析

### 3.2 查询缓存服务

**文件位置**：`src/module/database-optimization/query-cache.service.ts`

**核心方法**：

```typescript
/**
 * 获取缓存数据
 * @param {string} query - 查询语句
 * @param {any[]} parameters - 查询参数
 * @returns {Promise<T | null>} 缓存数据或null
 */
async get<T>(query: string, parameters?: any[]): Promise<T | null>

/**
 * 设置缓存数据
 * @param {string} query - 查询语句
 * @param {T} data - 缓存数据
 * @param {number} ttl - 过期时间(秒)
 * @param {any[]} parameters - 查询参数
 */
async set<T>(query: string, data: T, ttl?: number, parameters?: any[]): Promise<void>
```

**缓存键生成**：
```typescript
/**
 * 生成缓存键
 * 1. 规范化查询语句（去除多余空格，转小写）
 * 2. 将查询语句和参数组合
 * 3. 使用MD5哈希生成唯一键
 * 4. 添加配置的前缀
 */
private generateCacheKey(query: string, parameters?: any[]): string {
  const normalizedQuery = query.trim().toLowerCase().replace(/\s+/g, ' ');
  const paramsKey = parameters ? JSON.stringify(parameters) : '';
  const combined = `${normalizedQuery}:${paramsKey}`;
  const hash = crypto.createHash('md5').update(combined).digest('hex');
  return `${this.config.queryCache.keyPrefix}${hash}`;
}
```

**缓存策略**：
1. 先查询本地内存缓存（速度快）
2. 本地缓存未命中，查询Redis缓存
3. Redis缓存命中，更新本地缓存
4. 使用LRU策略淘汰最少使用的缓存条目
5. 定时清理过期缓存

### 3.3 读写分离服务

**文件位置**：`src/module/database-optimization/read-write-split.service.ts`

**核心方法**：

```typescript
/**
 * 获取数据库连接
 * @param {QueryContext} context - 查询上下文
 * @returns {DataSource} 数据库连接
 */
getConnection(context?: QueryContext): DataSource

/**
 * 获取Repository
 * @param {any} entity - 实体类
 * @param {QueryContext} context - 查询上下文
 * @returns {Repository<T>} Repository实例
 */
getRepository<T extends ObjectLiteral>(entity: any, context?: QueryContext): Repository<T>
```

**连接选择逻辑**：
```typescript
/**
 * 确定使用主库还是从库
 * 1. 如果强制指定，使用指定的连接
 * 2. 如果是事务操作，强制使用主库
 * 3. 否则根据读写比例随机选择
 */
private determineConnectionType(context?: QueryContext): ConnectionType {
  if (context?.forced) {
    return context.connectionType;
  }

  if (context?.transaction) {
    return 'master';
  }

  return this.shouldUseSlave() ? 'slave' : 'master';
}

/**
 * 根据读写比例决定是否使用从库
 * 默认70%的读操作走从库
 */
private shouldUseSlave(): boolean {
  const ratio = this.config.readWriteSplit.readWriteRatio;
  return Math.random() < ratio;
}
```

**事务管理**：
```typescript
/**
 * 在事务中执行操作
 * 1. 获取主库连接
 * 2. 创建QueryRunner
 * 3. 开启事务
 * 4. 执行回调函数
 * 5. 提交事务
 * 6. 异常时回滚事务
 * 7. 释放QueryRunner
 */
async withTransaction<T>(
  callback: (connection: DataSource) => Promise<T>
): Promise<T> {
  const context: QueryContext = {
    connectionType: 'master',
    forced: true,
    transaction: true,
  };

  const connection = this.getConnection(context);
  const queryRunner = connection.createQueryRunner();

  try {
    await queryRunner.connect();
    await queryRunner.startTransaction();

    const result = await callback(connection);

    await queryRunner.commitTransaction();
    return result;
  } catch (error) {
    await queryRunner.rollbackTransaction();
    throw error;
  } finally {
    await queryRunner.release();
  }
}
```

## 四、示例代码

### 4.1 使用慢查询监控

```typescript
import { SlowQueryMonitorService } from '@/module/database-optimization/slow-query-monitor.service';

@Injectable()
export class UserService {
  constructor(
    private readonly slowQueryMonitorService: SlowQueryMonitorService,
  ) {}

  async findUserById(id: number) {
    const startTime = Date.now();
    
    try {
      const user = await this.userRepository.findOne({ where: { id } });
      const duration = Date.now() - startTime;
      
      // 记录查询信息
      this.slowQueryMonitorService.recordSlowQuery(
        'SELECT * FROM user WHERE id = ?',
        duration,
        [id],
        'default',
        'User'
      );
      
      return user;
    } catch (error) {
      const duration = Date.now() - startTime;
      this.slowQueryMonitorService.recordSlowQuery(
        'SELECT * FROM user WHERE id = ?',
        duration,
        [id],
        'default',
        'User'
      );
      throw error;
    }
  }
}
```

### 4.2 使用查询缓存

```typescript
import { QueryCacheService } from '@/module/database-optimization/query-cache.service';

@Injectable()
export class ProductService {
  constructor(
    private readonly queryCacheService: QueryCacheService,
    private readonly productRepository: Repository<Product>,
  ) {}

  async findProductById(id: number) {
    const query = 'SELECT * FROM product WHERE id = ?';
    
    // 尝试从缓存获取
    const cached = await this.queryCacheService.get<Product>(
      query,
      [id]
    );
    
    if (cached !== null) {
      return cached;
    }
    
    // 缓存未命中，查询数据库
    const product = await this.productRepository.findOne({ where: { id } });
    
    // 将结果存入缓存，TTL为300秒（5分钟）
    await this.queryCacheService.set(
      query,
      product,
      300,
      [id]
    );
    
    return product;
  }
}
```

### 4.3 使用读写分离

```typescript
import { ReadWriteSplitService } from '@/module/database-optimization/read-write-split.service';

@Injectable()
export class OrderService {
  constructor(
    private readonly readWriteSplitService: ReadWriteSplitService,
  ) {}

  // 读操作：使用从库
  async findOrderById(id: number) {
    const repository = this.readWriteSplitService.getRepository(
      Order,
      { connectionType: 'slave' }
    );
    
    return await repository.findOne({ where: { id } });
  }

  // 写操作：使用主库
  async createOrder(orderData: CreateOrderDto) {
    const repository = this.readWriteSplitService.getRepository(
      Order,
      { connectionType: 'master' }
    );
    
    return await repository.save(orderData);
  }

  // 事务操作：自动使用主库
  async updateOrderStatus(id: number, status: string) {
    return await this.readWriteSplitService.withTransaction(
      async (connection) => {
        const queryRunner = connection.createQueryRunner();
        const order = await queryRunner.manager.findOne(Order, {
          where: { id }
        });
        
        order.status = status;
        await queryRunner.manager.save(order);
      }
    );
  }
}
```

## 五、可用场景

### 5.1 慢查询监控适用场景

1. **性能问题定位**：当系统响应变慢时，通过慢查询列表快速定位问题查询
2. **查询优化**：分析慢查询统计，找出需要优化的查询
3. **性能监控**：定期查看慢查询统计，监控数据库性能趋势

### 5.2 查询缓存适用场景

1. **热点数据查询**：频繁查询但很少变更的数据（如配置信息、字典数据）
2. **列表查询**：分页查询，特别是首页列表
3. **统计查询**：聚合统计类查询，计算结果相对稳定
4. **关联查询**：多表关联查询，结果集较大

### 5.3 读写分离适用场景

1. **读多写少场景**：如电商、社交、内容管理等系统
2. **报表查询**：数据分析、统计报表等只读操作
3. **实时性要求不高**：可以接受短暂的数据延迟
4. **主库压力大**：主库CPU、内存使用率高时启用

### 5.4 查询优化拦截器适用场景

1. **统一查询优化**：需要在多个Service中应用查询优化
2. **快速集成**：不想在每个Service中手动调用缓存和监控
3. **性能监控**：需要全面监控所有数据库查询性能

## 六、注意事项

### 6.1 慢查询监控注意事项

1. **阈值设置**：根据实际业务调整慢查询阈值，默认1000ms
2. **日志级别**：生产环境建议使用warn级别，开发环境使用info级别
3. **历史清理**：定期清理慢查询历史，避免内存占用过大
4. **查询参数**：记录查询参数时注意敏感信息脱敏

### 6.2 查询缓存注意事项

1. **缓存一致性**：数据更新时及时失效相关缓存
2. **TTL设置**：根据数据变更频率设置合理的TTL
3. **缓存大小**：控制缓存大小，避免内存占用过大
4. **缓存穿透**：对不存在的数据也要缓存空值，防止缓存穿透

### 6.3 读写分离注意事项

1. **主从同步**：确保主从数据库数据同步延迟在可接受范围内
2. **事务一致性**：事务操作必须使用主库，保证数据一致性
3. **读写比例**：根据实际读写比例调整，避免从库负载过高
4. **从库可用性**：从库不可用时自动降级到主库

### 6.4 查询优化拦截器注意事项

1. **选择性使用**：不是所有查询都需要缓存，根据业务场景选择
2. **缓存失效**：数据变更时及时失效相关缓存
3. **性能监控**：定期查看缓存命中率，调整缓存策略
4. **内存管理**：注意缓存大小，避免内存泄漏

## 九、性能优化建议

### 9.1 查询优化建议

1. **避免SELECT \***：只查询需要的字段，减少数据传输量
2. **合理使用索引**：为常用查询条件创建索引
3. **避免N+1查询**：使用JOIN或批量查询代替循环查询
4. **分页查询**：大数据量查询使用分页，避免一次性加载过多数据
5. **使用批量操作**：批量插入、更新代替单条操作

### 9.2 缓存优化建议

1. **热点数据预加载**：系统启动时预加载热点数据到缓存
2. **差异化TTL**：不同类型的数据设置不同的TTL
3. **缓存预热**：定时刷新热点数据，避免缓存雪崩
4. **监控缓存命中率**：定期查看缓存命中率，调整缓存策略

### 9.3 读写分离优化建议

1. **从库负载均衡**：多个从库时使用负载均衡
2. **读写分离策略**：根据业务特点选择合适的读写分离策略
3. **主从同步监控**：监控主从同步延迟，及时发现问题
4. **降级策略**：从库不可用时自动降级到主库
